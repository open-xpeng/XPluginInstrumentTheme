# 左侧导航卡片自动切换音乐卡片设计

## 背景

当前左侧子卡片由宿主 `InfoViewModel` / `SRInfoViewModel` 驱动：
- `0` 表示导航卡片
- `1` 表示音乐卡片

在实际使用中，左侧有时会停留在导航卡片位，但当前并没有导航引导内容，此时用户期望自动显示音乐卡片；一旦重新进入导航状态，则自动切回导航卡片。

## 目标

- 仅作用于左侧子卡片
- 全局生效，包括普通 `MainFragment` 和全屏地图 `MapFullFragment`
- 当原始左侧子卡片为导航卡片时：
  - 有导航引导内容时显示导航卡片
  - 无导航引导内容时显示音乐卡片
- 当原始左侧子卡片不是导航卡片时，不做任何干预

## 非目标

- 不修改右侧卡片逻辑
- 不改变宿主原始卡片选择策略
- 不重构宿主 `InfoViewModel` / `SRInfoViewModel` 数据结构

## 现状分析

宿主 APK 中：
- 左侧子卡片显示由 `InfoViewModel.onLeftCardIndex()` 推送到 `leftSubCardLiveData`
- `MainFragment` 使用 `InfoViewModel`
- `NaviSRFragment` 使用 `SRInfoViewModel`
- 导航相关状态由 `NaviViewModel` / `SRNaviViewModel` 提供，`getNaviTBtVisibility()` 与 `getNaviGuidenceVisibility()` 可用来判断当前是否存在导航引导内容

这说明我们不需要猜测导航状态，只需要在显示层对“左侧导航卡片”的最终显示结果做一次覆盖决策。

## 方案选择

采用“显示层覆盖”方案：

1. 保留宿主原始 `leftSubCardLiveData` 作为真实来源
2. 额外监听导航可见状态
3. 维护一份“原始左侧子卡片类型”和“当前是否在导航中”的组合状态
4. 只在原始类型为导航卡片时，决定最终显示导航卡片还是音乐卡片

不直接改写宿主 LiveData，也不直接篡改底层数据源，以减少副作用。

## 设计

### 状态模型

需要维护两类输入状态：

- 原始左侧子卡片类型
  - 来源：`InfoViewModel.leftSubCardLiveData`
  - 来源：`SRInfoViewModel.leftSubCardLiveData`
- 导航是否激活
  - 来源：`NaviViewModel.getNaviTBtVisibility()`
  - 来源：`SRNaviViewModel.getNaviTBtVisibility()`

最终规则：

- 若原始类型不为 `0`，直接显示原始类型
- 若原始类型为 `0` 且导航激活，显示 `0`
- 若原始类型为 `0` 且导航未激活，显示 `1`

### 挂点

在模块侧新增一个统一的左侧卡片切换协调逻辑，分别接入：

- `MainFragment` 左侧 SR/普通卡片显示链路
- `MapFullFragment` 左侧 SR 卡片显示链路

协调逻辑只负责算出“最终应该显示的左侧子卡片类型”，然后调用现有左侧卡片视图的 `showSubCardView()`。

### 数据流

1. 宿主继续推送原始左侧子卡片类型
2. 导航 ViewModel 推送导航可见状态
3. 模块协调层合并这两份状态
4. 左侧卡片视图按合并结果更新

### 错误处理

- 如果导航状态暂时不可用，按“未激活导航”处理
- 如果原始左侧子卡片类型为空或尚未初始化，不主动覆盖
- 若宿主后续把左侧切到其他卡片，立即让出控制

## 验证计划

重点验证以下场景：

1. 左侧原始卡片为导航卡片，当前无导航时，是否自动显示音乐卡片
2. 在无导航状态下启动导航，是否自动切回导航卡片
3. 结束导航后，是否自动恢复音乐卡片
4. 左侧切换到车辆、能耗、里程等非导航卡片后，是否不受影响
5. `MainFragment` 与 `MapFullFragment` 是否表现一致

## 风险

- 导航可见状态可能存在短暂抖动，需要避免重复无意义刷新
- 全局生效意味着普通页面和全屏地图页都要共享同一套判定逻辑，不能在两个页面分别写出不同规则
- 如果宿主未来修改 `0/1` 的卡片语义，这套覆盖规则需要同步调整

## 实施范围

本次只实现左侧全局自动切换逻辑，不包含右侧扩展，也不包含更复杂的用户自定义卡片策略。
